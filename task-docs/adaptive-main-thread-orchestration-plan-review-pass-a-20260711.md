## 2. Independently Reconstructed Outcome

The implementation plan must preserve these combined outcomes:

- adaptive main-thread governance becomes the default meaning of using the harness;
- full v2 remains available for explicit requests and stricter downstream policy;
- optional roles and stages do not weaken triggered safety, semantic, current-state, dirty-worktree, validation, or independent-evidence gates;
- task packets bind current work while static profiles retain durable intelligence and authority;
- the main thread owns topology, evidence arbitration, synthesis, intervention, final acceptance, and final reporting;
- retry controls cover stage-local quality failures, retry-domain cumulative quality failures, and artifact-free operational failures;
- no seventh fixed role, replacement fixed pipeline, or mandatory external orchestrator is introduced.

## 3. Current Conflicts And Gaps

The independent review identified these pre-plan gaps:

1. Fixed sequencing remained normative in the router, semantic protocol, orchestration prompt, plan-review PASS rule, routing scenario `RSM-005`, README, adoption runbook, minimal example, and several profiles.
2. No authority defined topology selection, delegation value, reviewer specialization, topology revision, evidence arbitration, cancellation, or synthesis.
3. No artifact defined mandatory delegation-packet fields, route/gate binding, default no-fan-out, or bounded nested-delegation budgets.
4. Delegated writes assumed `ACCEPTED_CONTRACT.md`; no lightweight stable packet boundary or versioning rule existed.
5. Retry behavior lacked stage-local and cumulative quality counters, operational-failure accounting, retry-domain identity, intervention, exhaustion, safety recovery, and controlled resume.
6. The run protocol lacked adaptive topology, branch state, dependency invalidation, retry state, and concise resumable context.
7. Calibration did not cover adaptive routing, wrong reviewer selection, evidence disputes, prompt failures, operational failures, intervention, or exhaustion.
8. The six profiles combined useful role intelligence with lifecycle assumptions, fixed handoffs, repo paths, or exhaustive schemas that belong in packets/templates.

## 4. Minimum Coherent Control Surfaces

The plan must cover, without defaulting to edits everywhere:

- control policy across router, adaptive/semantic protocols, and explanatory surfaces;
- delegation packets, route binding, no-fan-out, parallel ownership, stable boundaries, and proportional context;
- quality and operational retry state, retry domains, intervention, exhaustion, safety recovery, and controlled resume;
- optional templates and run artifacts for both lightweight and full governed paths;
- all six runtime profiles and adapter documentation;
- calibration and scenario evidence that tests adaptive behavior rather than renamed preset pipelines.

## 5. Non-Regression Invariants

- Triggered safety, secrets, current-state, semantic, dirty-worktree, validation, and high-impact confirmation gates remain mandatory.
- Original Request Anchor, Pass A/B, Outcome Contract, and AC-pass-but-user-fail controls remain when triggered.
- Evaluators remain read-only, producer-independent, evidence-first, and strict.
- Executors cannot reinterpret intent, expand scope, modify their authority, or repair outside it.
- HIGH-risk results require producer-independent verification distinct from main-thread final acceptance.
- Historical contracts, reports, QA, run directories, and the six role names remain understandable.
- LOW-risk work remains lightweight.

## 6. Dependency And Evidence Risks

- Partial migration would leave contradictory fixed-loop entry points.
- Runtime inheritance of router/routed docs was not proven; explicit packet binding is required unless runtime proof exists.
- Specialist review must map credibly onto the six profiles without multiplying mandatory roles.
- `max_depth = 1` does not alone enforce every nested-delegation budget.
- Router fixture validators prove string/path coverage, not semantic preservation or adaptive behavior.
- Documentation-only claims cannot prove live runtime enforcement; fixtures must label their evidence boundary.
