# Adaptive Orchestration Protocol

This is the normative control plane for Harness-managed work. Read it when task facts require topology selection or revision, a delegated capability, route/gate binding, a stable write Boundary, dependency control, intervention, retry, cancellation, synthesis, or producer-independent verification.

The Harness is one main-thread-controlled adaptive system. The main thread selects the smallest fact-required subset of optional capabilities and owns every invocation and transition. Capability availability, risk labels, Harness naming, prior sequence, or artifact availability never select a workflow.

## Capability Registry And Selection

The nine optional capabilities are independent registry entries, not stages:

1. Requirements Author
2. Requirements Evaluator
3. Planner
4. Plan Evaluator
5. Solution Designer
6. Solution Evaluator
7. Executor
8. Result Evaluator
9. Orchestration Reviewer

Before acting, main records the controlling result and non-goals, task kind, Operational Risk, Semantic Risk, current-state needs, uncertainty, protected state, dependencies, write conflicts, evidence needs, and triggered gates. It selects a capability only when that invocation adds task-specific production value, independent judgment, context isolation, or time-to-evidence value. Each selected capability receives a separate packet. Dependency facts determine invocation order for that Run.

There are no selectable mode labels, stored role sequences, parallel compatibility branches, or automatic all-capability path. Any subset is representable, including direct main work, a singleton, any mixed subset, or all nine. A Run that happens to select a previously familiar group is still an unnamed factual subset. HIGH risk requires complete triggered-control coverage, not every capability.

For LOW self-contained work with no gate or continuity trigger, main works directly, performs proportional validation, and creates no role invocation, Run tree, Context, Board, report, retrospective, or case. The word Harness and the existence of profiles do not create ceremony.

## Optional Capabilities, Mandatory Gates

Capabilities and durable artifacts are optional. Triggered gates are mandatory. Before every protected transition, main records an owner, primary evidence, and decision for each triggered safety, semantic, stable-Boundary, current-state, independence, retry, validation, and external/high-impact gate. Omitting a capability cannot omit the control commonly associated with it.

Always use AGENTS.md as the route table. Every invocation must either prove runtime inheritance of AGENTS.md plus the fact-triggered routed documents or list those routes and gates explicitly in its packet. Repository discoverability alone is not proof that a delegated context applied a rule.

## Main-Exclusive Authority

Only main may:

- reconstruct the user goal and select or revise topology;
- issue packets, launch an invocation, or cancel/supersede work;
- create, freeze, version, or amend an attempt Boundary;
- verify artifacts, hashes, protected inputs, and actual write scope;
- select and launch an evaluator or reviewer;
- classify failures and findings, authorize revision or retry, intervene, stop, unlock, transition, waive where allowed, or accept;
- arbitrate conflicting evidence and produce the user-facing synthesis.

Every capability is hard no-fan-out. An invoked capability must not create, invoke, launch, delegate to, or reconfigure another agent, even if its packet asks it to. It may return a question, clarification, escalation, artifact-ready notice, or evidence/finding reference only to main. Main issues a new packet and launches every later invocation.

Peer messages are non-authoritative. They cannot change scope, acceptance criteria, permission, topology, retry state, waiver, finding disposition, transition, unlock, or acceptance. System correctness must not depend on message delivery.

## Technical Solution Design And Attempt Boundary

Solution Design is selected only when uncertainty in the implementation path, affected surface, execution order, state/error/failure handling, irreversible points, or validation strategy materially affects reliable delivery. The Solution Designer produces one concrete Technical Solution Design candidate. The Solution Evaluator reviews exactly that Design.

Neither Solution capability creates, reviews, freezes, or implies a Boundary, permission artifact, Task result/scope/owner/dependency/acceptance change, lifecycle decision, or invocation. A predicted affected surface is not attempt permission.

Main derives the attempt-specific Boundary from the Frozen Task, accepted Design when present, current facts, protected state, and triggered gates. The Boundary may be:

- inline for simple direct work;
- carried by a delegated Task Packet; or
- persisted and versioned when continuity, repeated attempts, dirty/protected/shared state, high impact, or boundary drift makes recovery material.

Every form fixes the identity, objective, allowed actions and paths, protected state, required evidence and negative probes, validation, stop conditions, and material-change rule. A producer cannot edit or reinterpret its Boundary. A material change stops the attempt and requires a new main-issued identity/version.

New Runs do not select an execution-permission stage, permission-document loop, designer-owned permission path, or a renamed equivalent. Older Run permission artifacts remain immutable historical provenance only and cannot become new-Run authority.

## Producer Candidate Lifecycle

For a formal producer invocation, main:

1. selects the producer from facts and records its value;
2. assigns a unique invocation identity and one absent candidate path;
3. issues a packet with exact upstream identities/hashes, allowed reads, the candidate write, protected state, completion values, validation, stops, and hard no-fan-out;
4. records the changed-path baseline and critical upstream/protected hashes, and serializes unisolated writers;
5. launches the producer;
6. receives terminal identity/path/completion, computes the candidate hash, and verifies identity, completion, upstream/protected immutability, and actual write scope.

The producer writes only its candidate and cannot select or launch evaluation. Any extra or unattributable write, protected mutation, binding mismatch, or invalid completion makes the invocation invalid and prevents evaluation.

## Evaluator Report Lifecycle

Only after a valid candidate or result preflight may main:

1. select the matching evaluator/reviewer;
2. prepare the report parent, prove the invocation-specific report path is absent, and baseline target/upstream/protected hashes and changed paths;
3. issue an evaluator packet containing reviewer role, invocation identity, exact target path/identity/version/hash, exact upstream paths/hashes, one unique report path, prohibited surfaces, review type, canonical evaluator-calibration identity/path, completion values, error rules, and hard no-fan-out;
4. launch the evaluator, which directly reads exact primary target/upstream evidence and writes only its report;
5. after terminal, compute the report SHA-256 and verify required fields, report binding, target/upstream immutability, protected state, and actual write scope;
6. read the original report, classify findings, and alone decide revision, retry, intervention, stop, transition, unlock, or acceptance.

The evaluator runtime is truthfully labelled logical report-only write boundary plus post-execution verification. Report-capable workspace-write is not native path isolation, source/product write authority, runtime identity attestation, or permission to repair. The five evaluator/reviewer capabilities inspect primary evidence, issue strict PASS or FAIL, never repair or modify their target, and never start a producer, revision, or later invocation.

A complete report records reviewer role, invocation identity, target identity/version/hash, review type, inspected evidence, strict verdict, findings, completion state, calibration fields, unchecked areas, and a terminal self-hash boundary. The evaluator does not predict its final file hash. An optional main-owned receipt is derivative and never evaluator voice.

## Integrity Precedence And Report Recovery

Integrity and authority failures are decided before report content:

1. target/upstream/protected mutation, extra write, or unattributable delta: REVIEW_INVALID/STOP; do not consume verdict;
2. target identity/hash/version or upstream binding mismatch: REVIEW_INVALID/STOP;
3. collision, missing/unpreparable parent, directory-preparation failure, permission denial, runtime/tool failure, or no report after terminal: artifact-free operational failure; review incomplete/unavailable;
4. partial/unavailable completion, missing required fields, or no strict verdict: incomplete/invalid report;
5. complete verified strict FAIL: quality-failure evidence only after main validity classification;
6. complete verified strict PASS: evidence for main inspection only, never automatic transition or acceptance.

Observer timeout before terminal creates neither report completion nor quality failure. Missing review evidence is never fabricated from a transcript or summary. A collision is never overwritten and a partial report is never promoted.

One bounded same-cause operational retry is allowed only after main checks transience or corrects an already-authorized mechanical defect. Every retry uses a new invocation identity and new absent report path. Permission expansion requires explicit user confirmation before retry. A second same-cause failure requires intervention before a third attempt.

## Dependencies, Writers, Cancellation, And Synthesis

Do not start downstream work before required upstream evidence exists. Unisolated file-writing invocations are serialized. Parallel read-only work is used only for independent scopes with recorded owners, prerequisites, distinct outputs, shared-state restrictions, consumers, cancellation conditions, and main synthesis.

When new evidence changes a premise, risk, dependency, write set, Boundary, or validation path, main re-evaluates topology and cancels, supersedes, redirects, reorders, splits, combines, or replaces affected work. Superseded, cancelled, blocked, exhausted, partial, unavailable, invalid, or unverified output cannot flow downstream.

When evidence conflicts, main compares source authority, freshness, assumptions, reproducibility, and scope relevance. Agent count, majority vote, PASS labels, and confidence are not decision rules. Unresolved material ambiguity returns to the user or true owner.

## Failure Counters, Intervention, Exhaustion, And Resume

Maintain three separate finite counters:

1. stage-local valid quality failures for the current logical target/Boundary;
2. retry-domain cumulative valid quality failures across renamed stages, tasks, boundaries, roles, prompts, tools, and topologies;
3. artifact-free operational attempts keyed by normalized evidence-supported cause.

A quality failure counts only when a reviewable producer result and an evidence-supported in-scope controlling FAIL exist. Unsupported findings, wrong review type, evaluator misunderstanding, or artifact-free failure do not count as producer quality failures.

The first valid quality failure may receive one focused revision. A second stage-local or cumulative failure requires main intervention before a third attempt. Intervention inspects decisive producer/reviewer evidence, names the retry domain and counter, diagnoses likely origin, and changes a causally relevant input, Boundary, topology, reviewer, or strategy. Relabelling is not intervention.

A post-intervention third cumulative valid quality failure exhausts the retry domain. A post-intervention third same-cause operational failure blocks the domain when unavailable permission, service, environment, or external state controls progress; otherwise it exhausts that path. Automatic goal advancement and implementation writes stop.

Safety recovery is separate and requires already-scoped authority or precise user confirmation, its own Boundary, evidence, and applicable high-risk controls. Resume requires a substantive user decision or genuinely new independently verifiable evidence that changes the premise, with a finite budget of one producer/review cycle or one operational attempt. Rename, split, or a bare request to try again does not reset counters.

## Currentness, Verification, And Acceptance

An accepted Task result is bound to its exact primary evidence and affected-surface manifest, not a timeless repository identity. Later work that changes a behavior-shaping source or expectation names affected upstream Tasks/ACs, reruns their primary evidence, and receives a main-owned no-impact, reopen/reaccept, or supersession decision before downstream acceptance.

HIGH Operational Risk, HIGH Semantic Risk, and high-impact results require producer-independent verification of applicable primary outcome, safety, semantic, and state evidence. HIGH Semantic Risk also requires independent semantic judgment. Producer self-report and producer-run checks are insufficient.

Main always owns final acceptance. It directly inspects proportional primary evidence, including producer-independent Result QA when triggered, resolves independent findings, verifies protected/write scope, and reports what changed, what was not changed, what was validated, what remains unvalidated, blockers, and residual uncertainty. Passing tests are evidence, never proof of all real-world behavior.

## Complete-Run Learning Control

After one non-trivial requirement Run physically reaches exactly `accepted`, `blocked`, `exhausted`, or `stopped`, main performs one lightweight source-controlled scan over the current terminal observation and evidence index. Task completion is not a Run terminal event. A clean scan creates no formal retrospective, case candidate, learning directory, agent, Loop, Task, or enhancement.

The scan identity binds the Run identity, terminal observation, terminal evidence index, and current learning-contract digest; invocation identity and wall-clock time do not mint another scope. When source persistence selects Context, Context alone stores the canonical record in `failureScan`. Otherwise main holds the exact canonical inline record in the uninterrupted terminalization session; Board and Task State remain reference inputs and never own or emit scan truth. Exact prior bytes no-op, conflicting or stale bytes stop, and an already-terminal fresh session without exact prior scan history returns `SCAN_HISTORY_UNAVAILABLE_STOP` rather than pretending to perform a first scan.

A planned later invocation, handoff, compaction, restart, or main-session end with continuing work creates continuity risk. Before leaving the session, main recomputes persistence, transfers the exact unchanged inline scan bytes to a newly selected Context through normal Task State → Board → Context publication, and reads them back. The transfer performs no scan, does not rewrite identity, does not add a scan field to Board or Task State, and later reentry no-ops. Unexpected loss before transfer is unavailable; native crash recovery is not claimed.

The exact trigger projection is source-owned and covers all seventeen Requirements learning alternatives. Main fact-selects the existing Result Evaluator only for a `run-learning-case-review` packet after a complete evidence-backed candidate exists. The reviewer writes one unique report only; it cannot repair, promote, launch, transition, or accept. Main dispositions review evidence but never impersonates the later human case disposition.

No retrospective, candidate, review, main disposition, or central case authorizes a Harness change. A direct request to mutate source, calibration, protocol, template, profile, validator, test, package, topology, agent lifecycle, Task state, or enhancement state; emit a global `MUST`; or start work stops with `AUTOMATIC_LEARNING_CHANGE_STOP`. Even reviewed case evidence plus a later human decision, new Task, and Boundary returns only a route to a separately selected Task; the learning decision itself may neither write nor launch.

## Readiness, Currentness, And Verification-Only Topology

Readiness is recomputed from the exact current authority and Task identities, named owner and consumer, completed prerequisites with accepted current evidence, absence of blocker/conflict/pending choice, closed triggered gates, and one authorized next action. Downstream unlock is a separate main decision that binds predecessor evidence and downstream readiness. Blocked, exhausted, superseded, cancelled, partial, unavailable, invalid, waived, stale, or evaluator-only PASS output cannot unlock.

Integrity and authority faults outrank content. Authority or Frozen-Task mismatch, missing owner/consumer, protected or extra write, duplicate truth owner, identity/currentness conflict, illegal transition, missing physical observation, and incomplete output stop before a quality verdict. A complete strict FAIL or PASS remains evidence for main; neither automatically transitions or accepts.

Task 3 currentness binds the current owner, material inputs, primary evidence, affected upstream Task/AC set, downstream consumer revalidation, and a main no-impact/reacceptance disposition. Missing or stale input returns `STALE_OWNER_STOP`; a generic green command cannot close it. Blocked, superseded, or cancelled predecessor state and evaluator-only unlock fail through the separate readiness decision.

Topology is recomputed from exact current facts and the exact remaining write set. If the write set is empty, the source-derived capability identity excluded by the decision must resolve to exactly one current registry entry, verification runs without that Executor, and its sentinel manifest remains unchanged. If a write or premise change appears, verification stops without mutation, invalidates the old input digest, and returns to main for a new topology and Boundary. It never silently adds an Executor or repairs state.

Retry decisions consume physically read-back records whose stage, Boundary lineage, retry domain, observable result, normalized cause authority, and prior history-snapshot identities remain stable. Counts must be monotonic against that bound history. A rename, split, role/tool/topology change, or state rewrite is display metadata and cannot mint a first failure or lower a counter. Missing, mismatched, or discontinuous identity/history stops before retry selection. The first valid quality failure permits one focused revision; the second requires intervention before a third; a post-intervention third exhausts. The first same-cause operational failure permits one checked retry; the second requires intervention before a third; a post-intervention third blocks or exhausts the path. Resume from exhaustion requires a substantive user decision or genuinely new independently verifiable evidence plus a one-cycle or one-attempt budget.

The adjacent records are the executable projection of this protocol. Templates may reference them but cannot redefine readiness, currentness, retry, error precedence, or topology. A prose/record mismatch is a strict semantic failure.
