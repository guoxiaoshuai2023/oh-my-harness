# Run Directory Protocol

Use this protocol only when durable evidence, cross-context continuity, versioned authority, retry history, or independent review is factually required. A simple self-contained task stays inline and creates no Run directory, packet file, Context, Board, report, retrospective, or case merely because the Harness is used.

## Proportional Artifact Decision

- Inline main-thread control is sufficient for LOW direct work without continuity or review triggers.
- A concise Task Packet carries a delegated invocation Boundary when its fixed identity, objective, writes, protected state, evidence, and stops are sufficient.
- A persisted/versioned Boundary is required when continuity, repeated attempts, dirty/protected/shared state, high-impact action, material ambiguity, or boundary drift makes recovery material.
- Candidate, evaluator report, implementation report, and independent QA artifacts are created only when selected by topology or required by a triggered evidence/independence gate.
- Triggering one artifact never creates an empty directory skeleton.

Older Run artifacts remain byte-preserved historical provenance. They may be read for their original Run but do not enter the authority chain or select behavior for a new Run.

## Namespace Ownership

- Main owns packets, Boundaries, receipts, dispositions, accepted/superseded transitions, Context, Board, Task state, topology, retry/intervention, unlock, and acceptance.
- An assigned producer writes one exact candidate or producer-report path.
- An assigned evaluator/reviewer writes one exact invocation-specific report path.
- Candidate and report paths must be absent before launch and are never reused or overwritten.
- Evaluator terminal report bytes are immutable. Main computes the final report hash afterward.
- Directory placement never grants authority. Every controlling artifact records type, identity/version, status/completion, owner, exact upstream bindings, and supersession rule.

## Candidate And Report Exchange

Main selects the producer, issues a packet, records candidate/upstream/protected baselines, launches, and verifies candidate identity/hash/completion and actual write scope. Only after valid verification may main select and launch the evaluator.

Before evaluator launch, main prepares the report parent, proves the report path absent, and records target/upstream/protected hashes plus changed paths. The evaluator directly reads the exact candidate/result and upstream, writes only its report, and returns terminal identity/path/completion. Main then computes the report hash and verifies binding, target/upstream immutability, protected state, and exact changed scope before reading the verdict or dispositioning findings.

Evaluator workspace-write is a logical report-only write boundary plus post-execution verification, never native path isolation or product/source authority. A report contains role, invocation, target identity/version/hash, review type, primary evidence, strict PASS or FAIL, findings, completion, calibration fields, unchecked areas, and a terminal self-hash boundary. A main receipt is optional derivative evidence and never impersonates evaluator voice.

## Error Precedence And Recovery

Decide integrity before verdict:

1. target/upstream/protected mutation, extra write, or unattributable delta: REVIEW_INVALID/STOP;
2. target or upstream binding mismatch: REVIEW_INVALID/STOP;
3. collision, missing/unpreparable parent, directory preparation, permission, runtime/tool, or terminal missing-report failure: operational incomplete/unavailable;
4. partial/unavailable or malformed report: report incomplete/invalid;
5. verified complete strict FAIL or PASS: quality evidence for main disposition, never an automatic transition.

Never invent a missing report from transcript, summary, or receipt. Never promote partial output or overwrite a collision. One bounded operational retry requires a checked transient or corrected authorized defect and a new invocation plus new absent report path. Permission expansion requires explicit user confirmation. The second same-cause failure requires intervention before a third.

## Boundary Identity And Material Change

The attempt Boundary may be inline, packet-carried, or persisted/versioned. Every form fixes identity, objective, allowed actions and paths, protected state, required evidence, validation, stops, and material-change rule. The producer cannot change it.

A material change to objective, scope, write set, protected state, evidence, route/gate binding, dependency, confirmation, or stop condition ends the attempt. Main issues a new identity/version only after returning the change to its true owner. Older Run permission documents remain historical only; new Runs use main-issued Boundary semantics.

## Continuity, State, And Compaction

When continuity is triggered, record goal/non-goals, current authority, topology/gates, sourced facts/freshness, decisions, assignments/dependencies, protected state, evidence, retry counters, findings/dispositions, intervention, stop/resume state, blockers, and one next action. Distinguish pending, active, blocked, exhausted, completed, superseded, and cancelled.

Compaction may shorten prose but cannot remove failures, valid/unsupported findings, retry/exhaustion, intervention, waiver, rejected/pending decisions, blockers, or exact evidence identities. Accepted authority prevails over conflicting state; automatic advancement stops until main repairs the conflict.

### Recoverable Work-Truth Decisions

Persistence is selected independently from facts, never from an artifact count. No continuity, multi-task/wrong-unlock, or detailed evidence/retry trigger keeps state inline and creates no Run tree. Continuity creates Living Context; multi-task dependency, blocker, consumer-readiness, retry-summary, or wrong-unlock risk creates Task Board; full AC/evidence, attempt/finding, waiver, intervention, or retry detail creates per-task state. Triggered views may coexist, but an absent view is not fabricated and no trigger forces an empty skeleton.

Main is the sole writer for inline, Context, Board, and Task State truth. Context owns Run macro truth, Board owns the task index and readiness, and Task State owns full evidence and retry detail. A wrong writer, duplicate controlling value, identity/hash mismatch, unsupported Board readiness, or partial multi-view publication stops. The last fully verified identities remain current.

The lifecycle states are exactly `pending`, `active`, `blocked`, `exhausted`, `completed`, `superseded`, and `cancelled`; `accepted` is an event or binding, not a state. Only an observed main actor may use a permitted edge. `active -> completed` is a distinct decision: it requires physically read-back current-behavior and negative evidence, a valid strict-QA artifact, finding disposition, and a main-acceptance artifact, all with stable observation identities. Other legal edges remain distinct; every unsupported or incompletely observed completion edge stops. Downstream unlock is a separate adaptive-protocol decision.

For durable publication, first consume the source-owned persistence result, omit `inline`, and map only its durable representations to participating views. Filter the full relative order owned by `publication.current` to that participating set; do not create a nonparticipating placeholder. Construct and hash every selected next view before writing, then stage, publish, and read back the selected Task State, Board, and Context views in that source-derived relative order. Every participating view binds one update identity and its expected hash. Board binds Task State only when both participate, and Context binds Board only when both participate. Read and validate the physical prior last-verified record before mutation. Only the last participating view becomes the physical commit root after every readback matches and the exact allowed manifest delta is observed. A protected or extra delta, wrong order/set, conflicting applicable cross-view reference, corrupt readback, missing view, partial shared update, or missing/corrupt prior recovery record is never current. Recovery returns only the identity parsed from the observed prior record; it cannot invent a literal. Native cross-file atomicity is not claimed.

Task 2 completion requires physical evidence: source-authorized ownership, an attributable disk delta restricted to the allowed set, exact byte readback with an observation identity, and no Board/Task conflict. A rejected wrong writer must produce zero delta; a bypass, leakage, unobserved/corrupt result, or conflicting physical view stops. An object or event-log claim is not observation.

Every waiver is a physically read-back record with its own identity and exact gate, authority/owner, existing-or-missing evidence state, reason, scope, issue time, expiry/re-evaluation, residual risk, downstream impact, consumer revalidation, and status fields. A completeness or authority boolean cannot substitute for those fields. Authorization, secrets/safety, destructive confirmation, protected state, no-fabrication, independent PASS, and main acceptance are non-waivable. A waiver never becomes PASS or unlock.

Retry storage physically binds a stable stage identity, Boundary lineage, retry-domain identity, observable-result identity, normalized root-cause authority identity, and prior history-snapshot identity. It keeps stage-local valid quality, cumulative retry-domain quality, and normalized-cause operational attempts separate and monotonic. Unsupported findings and observer timeouts do not increment quality. Stored counter identity and history survive renames, task splits, roles, tools, topology changes, and state rewrites; discontinuity or a lower rewritten count stops. The adaptive protocol alone decides focused revision, intervention, exhaustion, and resume.

### Bottom-Up Complete-Run Learning

Formal learning is bottom-up and trigger-created. A completed scan with no source trigger creates no `learning/` directory. A trigger plus sufficient privacy-safe evidence creates exactly one retrospective at `task-docs/history/<run>/learning/RETROSPECTIVE-<version>.md` and exactly one candidate at `task-docs/history/<run>/learning/cases/candidates/NEGATIVE_CASE-<version>.md`; no review, receipt, or disposition branch exists until that artifact is separately authorized. Partial or colliding paths are never overwritten. The retrospective and candidate carry exact source Run/end/scan/evidence/authority identities, facts, timeline, replay, actual failure signal, evidence-backed root cause and uncertainty, applicability/generalization/non-applicability, duplicate boundary, privacy, proposed owner/route, expected side effects, and explicit no-authority declarations.

The source root-cause layers are exactly requirements, architecture, plan/ownership, design, boundary, evidence/validator, implementation, orchestration/topology, operational/runtime, and review-mismatch. One primary and zero or more contributing layers require evidence; unresolved causes remain unresolved. Retrospectives and candidates contain safe evidence references, not raw sensitive material, and cannot declare a global rule, promotion, permission, package effect, implementation choice, or launch instruction.

An independently reviewed per-run candidate still cannot mutate the central source library. Main may route it for human consideration only. A later explicit human disposition and a named human-authorized case custodian must bind the exact review, main route, target/preimages, complete library manifest, allowed semantic changes, and protected/package zero-delta baseline before any central action. T-03 evidence uses only synthetic controlled temporary roots and never writes the real `task-docs/cases/` path.

Central statuses are `active`, `downgraded`, `merged`, `superseded`, and `retired`; rejection has no central postimage. Create adds one active v1 case. Update preserves identity/path/status, increments version, changes only human-enumerated fields, and preserves prior consumption history. Merge replaces every named active/downgraded predecessor as merged with reciprocal successor links and adds one active v1 successor with sorted `mergedFrom`; content and history are not silently lost. Downgrade replaces one active case with a narrower applicability boundary, `downgraded` status, and `reference-only` consumption without changing facts/evidence/replay. Reject produces zero central delta. Supersede replaces one active/downgraded predecessor as closed `superseded` and adds one reciprocal active v1 successor. Retire replaces one active/downgraded case as `retired`, preserves content/history, closes new consumption, and never deletes.

Every current central postimage includes stable identity, positive version, status, source Runs, semantic content identity, complete LEARN-06 facts/evidence/replay/failure/root-cause/applicability content, review/human/custodian/source-contract provenance, explicit predecessor/successor/merge links, and immutable downstream-consumption history. Containment/writer faults outrank missing human custody; custody faults outrank stale/conflicting preimages; preimage faults outrank semantic faults. All pre-publication stops produce zero delta. Exact staged bytes and full expected manifest must match physical readback; a false postimage is invalid even when action, actor, and path are correct. Multi-path fixture publication claims no native atomicity; an invalid disposable fixture is discarded rather than promoted.

## Dirty Worktree And Writer Serialization

Record protected read-only paths, pre-existing user work, allowed deltas, and before/after hashes or changed-path inventories when a final diff cannot prove attribution. Preserve untracked and ignored work. Unisolated writers are serialized; read-only parallel work needs distinct scope, prerequisites, outputs, shared-state restrictions, consumer, cancellation condition, and main synthesis.

## Reports, Independent QA, And Acceptance

Producer reports cite the exact packet/Boundary and state result, changed scope, protected-state proof, evidence, validation, unvalidated areas, blockers, and uncertainty. They never claim independent QA or main acceptance.

Independent QA reads actual state and primary evidence, uses the assigned review type, writes only its unique report, and returns strict PASS or FAIL. Main validates report/target/upstream/write scope, dispositions findings, and alone transitions or accepts. Missing or invalid evidence never unlocks downstream work.
